## A1.6 — Agent Person schema fix + blog/report author decision (20 minutes)

**Context:** There are two correctness problems in the current schema:

1. Agent pages at `/agents/[slug]` have a Person schema with `worksFor: Montaic`. This is factually false — agents are *customers*, not employees. Every major AI engine treats `worksFor` as an employment assertion. Having hundreds of independent agents declared as Montaic employees dilutes the brand entity and is a schema integrity violation.

2. Blog posts and market reports currently have an `author` field that A1.5 deliberately left untouched pending a decision. The decision is now made: **Lance Roylo as founder-author on blog posts, "Montaic" as publisher on market reports** (since those are programmatically generated).

### A1.6.1 — Fix agent Person schema

**File:** `app/agents/[slug]/page.tsx` (or wherever the agent microsite page component lives)

Find the existing Person JSON-LD block. It currently looks something like:

```ts
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": agent.name,
  // ...other Person fields...
  "worksFor": { "@type": "Organization", "name": "Montaic" }  // ← DELETE THIS
}
```

**Replace the entire schema block with this pattern:**

```ts
{
  "@context": "https://schema.org",
  "@graph": [
    {
      "@type": "ProfilePage",
      "mainEntity": {
        "@type": "Person",
        "name": agent.name,
        "description": agent.bio || agent.description || undefined,
        "image": agent.photo || agent.image || undefined,
        "url": `https://montaic.com/agents/${agent.slug}`,
        "jobTitle": agent.title || "Real Estate Agent"
      },
      "url": `https://montaic.com/agents/${agent.slug}`,
      "publisher": { "@id": "https://montaic.com/#organization" }
    }
  ]
}
```

**What changed:**
- Removed `worksFor` entirely (false claim eliminated)
- Removed `affiliation` if present (same problem — implies an org relationship that doesn't exist)
- Wrapped Person in a `ProfilePage` with `mainEntity` (Montaic publishes the page but the agent is the subject)
- Added `publisher` pointing to the root Organization `@id` (correct: Montaic publishes the profile, but doesn't employ the agent)

**Important:** Adapt the field names (`agent.name`, `agent.bio`, `agent.slug`, etc.) to match whatever variable names the existing code uses. The pattern above is the target shape; the actual prop names might be `agentData.fullName` or similar in your codebase.

If agent pages also have any `"sameAs"` links to agent social profiles or brokerage pages, keep them inside the Person object — those are correct and helpful.

## A2 — BreadcrumbList across all page types (40 minutes)

**Context:** Montaic currently has zero breadcrumbs across 222 URLs, despite having a clean hierarchical taxonomy (`home > tools > mls-description-generator`, `home > markets > austin-tx`, etc.). BreadcrumbList is a rich result in Google SERPs and a primary structural signal for AI engines understanding site hierarchy.

### A2.1 — Create the breadcrumb utility

**File:** `lib/schema.ts` (or wherever schema utilities live — may already exist from A1/A1.5 work)

Add this utility function:

```ts
type BreadcrumbItem = {
  name: string;
  path: string;
};

const BASE_URL = "https://montaic.com";

const SEGMENT_LABELS: Record<string, string> = {
  tools: "Tools",
  markets: "Markets",
  compare: "Compare",
  blog: "Blog",
  marine: "Marine",
  ports: "Ports",
  "vessel-types": "Vessel Types",
  "listing-generator": "Listing Generator",
  agents: "Agents",
  pricing: "Pricing",
  "free-listing-generator": "Free Listing Generator",
  "fair-housing-compliance-checker": "Fair Housing Compliance Checker",
};

function titleize(slug: string): string {
  if (SEGMENT_LABELS[slug]) return SEGMENT_LABELS[slug];
  return slug
    .split("-")
    .map((w) => w.charAt(0).toUpperCase() + w.slice(1))
    .join(" ");
}

/**
 * Generate BreadcrumbList JSON-LD from an explicit list of crumbs.
 * "Home" is prepended automatically.
 */
export function breadcrumbSchema(items: BreadcrumbItem[]) {
  const crumbs = [{ name: "Home", path: "" }, ...items];
  return {
    "@type": "BreadcrumbList",
    itemListElement: crumbs.map((c, i) => ({
      "@type": "ListItem",
      position: i + 1,
      name: c.name,
      item: `${BASE_URL}${c.path ? "/" + c.path : ""}`,
    })),
  };
}

/**
 * Auto-generate BreadcrumbList from a URL path.
 * Works for ~80% of pages. Use breadcrumbSchema() for overrides.
 * @param path  e.g. "tools/mls-description-generator"
 * @param finalName  optional override for the last crumb's display name
 */
export function breadcrumbSchemaFromPath(path: string, finalName?: string) {
  const segments = path.split("/").filter(Boolean);
  const items: BreadcrumbItem[] = segments.map((seg, i) => ({
    name:
      i === segments.length - 1 && finalName ? finalName : titleize(seg),
    path: segments.slice(0, i + 1).join("/"),
  }));
  return breadcrumbSchema(items);
}
```

### A2.2 — Wire into each page type

The strategy: use `breadcrumbSchemaFromPath()` for most pages (auto-generates from URL), and explicit `breadcrumbSchema()` for any page where the auto-titleize produces a bad name.

**For pages that already have a `@graph` array in their JSON-LD** (like the homepage from A1), add the breadcrumb object to the existing `@graph`:

```ts
import { breadcrumbSchemaFromPath } from "@/lib/schema";

// Inside the existing @graph array, add:
breadcrumbSchemaFromPath("tools/mls-description-generator", "MLS Description Generator")
```

**For pages that have a standalone JSON-LD block** (no `@graph`), add a separate `<script type="application/ld+json">`:

```tsx
<script
  type="application/ld+json"
  dangerouslySetInnerHTML={{
    __html: JSON.stringify({
      "@context": "https://schema.org",
      ...breadcrumbSchemaFromPath("tools/mls-description-generator", "MLS Description Generator")
    })
  }}
/>
```

### A2.3 — Layout-level approach (optional shortcut)

If the codebase has a shared layout component that wraps all marketing pages, you can add breadcrumbs at the layout level using `usePathname()` and a mapping function. This covers all 222 URLs in one shot:

```tsx
// In the shared layout or a BreadcrumbSchema component
"use client";
import { usePathname } from "next/navigation";
import { breadcrumbSchemaFromPath } from "@/lib/schema";

export function BreadcrumbSchema({ finalName }: { finalName?: string }) {
  const pathname = usePathname();
  if (pathname === "/") return null; // Homepage doesn't need breadcrumbs
  const path = pathname.startsWith("/") ? pathname.slice(1) : pathname;
  const schema = {
    "@context": "https://schema.org",
    ...breadcrumbSchemaFromPath(path, finalName),
  };
  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(schema) }}
    />
  );
}
```

Then on each page: `<BreadcrumbSchema finalName="MLS Description Generator" />`

Or with no `finalName`, the auto-titleize handles it. The `SEGMENT_LABELS` lookup table in the utility covers the common cases. Anything not in the table gets title-cased from the slug.

**Tradeoff:** The layout approach is faster to ship but produces auto-titleized names that may be imperfect (e.g., "Austin Tx" instead of "Austin, TX"). The per-page approach lets you pass the exact display name. Recommendation: ship the layout approach first (covers 222 URLs immediately), then override the worst auto-names in a follow-up pass.

## Known risks

- **Agent page variable names:** The Person schema fix in A1.6.1 uses placeholder variable names (`agent.name`, `agent.slug`). Adapt to whatever the actual codebase uses.
- **`SEGMENT_LABELS` gaps:** The auto-titleize function may produce imperfect names for segments not in the lookup table. This is fine for v1 — add missing labels as you encounter them.
- **`@graph` merging:** If a page already has a `@graph` array (homepage, free-grader from A6), add the BreadcrumbList object to the existing array rather than creating a second `<script>` tag. Two separate `<script>` tags also work, but one `@graph` is cleaner.
- **`usePathname()` is a client component hook.** If using the layout-level approach (A2.3), the `BreadcrumbSchema` component needs `"use client"` directive. The JSON-LD still renders in the HTML for crawlers because Next.js SSR-renders client components on first load.
