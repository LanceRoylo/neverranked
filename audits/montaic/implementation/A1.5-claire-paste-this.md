## What this does

A1 added Organization + WebSite schema at the root layout. Three pages on Montaic already have their own per-page schema blocks, but they were written before the root Organization existed and don't reference it. They're orphaned schema islands.

A1.5 fixes that. Each per-page schema gets a small `publisher` reference that points back to the root Organization via `@id`. This converts disconnected islands into a connected entity graph that AI engines and Google can traverse — which is exactly the structure that makes per-page content "count" toward the Montaic entity instead of floating disconnected.

**Three files. One added property each. No structural changes.**

## Step 1 — Patch SoftwareApplication on the pricing page

**File:** `app/(marketing)/pricing/page.tsx`

Find the existing `JSON_LD` constant (or whatever the SoftwareApplication schema is named — Claire, you reported this file already has a JSON_LD pattern).

**Add this property** to the top-level SoftwareApplication object:

```ts
"provider": { "@id": "https://montaic.com/#organization" }
```

**Where exactly:** Add it as a sibling of `"name"`, `"applicationCategory"`, `"offers"`, etc. — anywhere in the top-level object is fine.

**Why `provider` and not `publisher`:** For SoftwareApplication schema, `provider` is the canonical Schema.org property for "the entity that provides this software." `publisher` is also valid but `provider` is more semantically correct for a SaaS product.

**Do not change anything else** in the existing SoftwareApplication block. Don't refactor offers, don't add aggregateRating, don't touch the price formats. Surgical addition only.

## Why we're NOT touching the Person schema on `/agents/[slug]`

You flagged that the agent microsite pages at `app/agents/[slug]/page.tsx` have a Person schema. The instinct is to add `"worksFor": { "@id": "https://montaic.com/#organization" }` to it.

**Don't.** Real estate agents on Montaic are *customers*, not employees. Declaring `worksFor: Montaic` would be factually false and would dilute Montaic's entity signal by associating the brand with hundreds of independent agents. Same problem with `"affiliation"` — implies an organizational relationship that doesn't exist.

The right pattern for agent pages is to add a separate `ProfilePage` or `WebPage` schema that references the Person as `mainEntity` and the Organization as `publisher` (because Montaic publishes the page even though the agent is the subject). That's a meaningful schema architecture decision and belongs in its own ticket — flag it as **A1.6**, do not address it in A1.5.
