# State of Agent Readiness — monthly cadence

The agent-readiness panel is intended to grow over time, with monthly re-runs against an expanding site list. This document is the operating manual.

## Cadence

**Monthly, first Sunday of each month, 14:00 UTC.** Runs after the daily citation cron at 06:00 UTC has finished and the State of AEO weekly has shipped (Friday). Gives the dataset its own weekly slot where nothing else competes.

## How it runs today (manual)

Until we wire this to a worker cron, the monthly run is manual:

```bash
cd /Users/lanceroylo/Desktop/neverranked
node scripts/agent-readiness-test-harness.mjs
```

Output:
- `data/agent-readiness-baseline-{YYYY-MM-DD}.json` — full structured dataset
- stdout — per-site scan summary + aggregate findings

After the scan, the dataset gets committed to the repo so the history is versioned:

```bash
git add data/agent-readiness-baseline-{YYYY-MM-DD}.json
git commit -m "agent-readiness baseline: {month-name} 2026 panel scan"
git push
```

## How it should run eventually (Cloudflare worker cron)

A future enhancement: deploy this as a separate Cloudflare worker with a cron trigger. Same pattern as the existing citation worker.

Sketch:

```jsonc
// wrangler.jsonc for a new mcp-server-cron worker
{
  "name": "agent-readiness-monthly",
  "main": "src/cron.ts",
  "triggers": {
    "crons": ["0 14 1-7 * 0"]  // first Sunday of each month, 14:00 UTC
  }
}
```

The worker would:
1. Read the panel from a stored config (could live in R2 or be inline)
2. Run `agent_readiness_check` against each URL
3. Write the dataset to R2 with a date-stamped key
4. POST a summary to a webhook or Slack channel for human review

Estimated worker compute: roughly 91 sites × 1-2 fetch calls × 2-5 second average = 5-15 minutes of total CPU. Well within Cloudflare worker time limits if each scan runs as its own workflow step (similar pattern to the citation cron's per-keyword dispatch).

## Site list governance

The panel is currently inline in `scripts/agent-readiness-test-harness.mjs`. Should be moved to a separate config file (`data/agent-readiness-panel.json` or similar) so:
- The site list can grow without code changes
- Diffs are easier to read
- Future contributors can suggest sites by PR

When adding sites, follow these rules:
1. **Use root URL only.** No deep links. The methodology measures root-URL deployment; subpage crawling is a future v3.
2. **Specify a vertical.** Use one of the 13 supported verticals (see `mcp-server/src/tools/agent-readiness-check.ts` VERTICAL_BASELINES). Use `null` only for controls.
3. **Use the brand's actual canonical URL.** No tracking params, no www-vs-not redirects. Test the URL responds 200 before committing.
4. **Document why.** When adding a new site, note which vertical bucket it fills (existing or expanding). Avoid adding 5 banks when banking is already over-represented.

## How the dataset gets used

1. **State of Agent Readiness report:** the seed report at `reports/state-of-agent-readiness/` re-renders from the latest dataset. Monthly cadence produces 12 reports per year.
2. **Audit section 03b:** every audit generated by `audit-generate.mjs` runs `agent_readiness_check` against the client URL. The dataset provides the "how you compare" context.
3. **Cold outreach:** the cover email templates at `content/outreach/state-of-agent-readiness-cover-emails.md` reference the dataset findings.
4. **Eventual public publication:** when the panel is 250+ sites and we have 6+ months of history, the data becomes a public report. Until then, private outreach asset only.

## Known limitations

- **Root URL only.** Actual deployment may be higher than this scan reflects. ~5% finding is a floor.
- **Bot blocking.** ~25% of major sites return 403 to an automated request. The block rate is itself a finding.
- **Single fetch per site.** No retries, no rendering with JS. Some SPA sites may have ActionTypes injected client-side that we miss.
- **No screenshots.** We see what an agent doing an HTML fetch sees. A real agent using a headless browser would see more.

These limitations are documented in the methodology section of every report so readers know what the number does and doesn't claim.

## Failure modes

- **Harness crashes mid-run:** results are written at the end, so a crash loses the run. Future: write per-site results as they land.
- **All sites blocked:** if blocking detection improves industry-wide, the panel becomes scan-resistant. Mitigation: maintain a private fallback panel of sites we've gotten consent from, OR move to a residential-IP proxy.
- **Network from your laptop is slow:** the scan takes 5-15 min instead of 2-3. Acceptable.

## When to re-version the methodology

Bump the `detection_version` field in the dataset (currently `v2`) when:
- ActionType vocabulary changes (new types added or deprecated)
- Detection format support changes (e.g., add RDFa, JSON-LD subpage crawl)
- Scoring weights change
- Vertical baselines change

The current version is `v2`. The next major bump will be `v3` when subpage crawling lands.
